# Vibe coding

本章节完全由 AI 生成。

## 什么是 Vibe coding？

> "You just give in to the vibes, embrace exponentials, and forget that the code even exists."
>
> — Andrej Karpathy, 2025

**Vibe coding** = 用自然语言描述需求，让 AI 生成代码，你负责审查、调试和把控方向。

不是"摆烂让 AI 写"，而是一种**人机协作的新范式**——你从"写每一行代码"转变为"做技术决策 + 审查生成结果"。

## 工具生态

| 工具 | 定位 |
|------|------|
| **GitHub Copilot** | IDE 内的 AI 补全 + Chat，最主流的日常工具 |
| **Cursor** | 整个 IDE 围绕 AI 设计，深度集成多模型 |
| **Claude Code** | CLI 工具，适合终端工作流和自动化 |
| **v0 / Bolt** | 自然语言生成前端页面/组件，适合快速原型 |
| **ChatGPT / Claude** | 通用对话，适合分析需求、解释代码、设计架构 |

选择哪个不重要，重要的是**理解它能做什么、不能做什么**。

## 实践方法

一个典型的 Vibe coding 工作流：

```
1. 明确需求：我要实现什么功能？有哪些约束？
2. 拆分任务：把大需求拆成小的、具体的步骤
3. 描述给 AI：用清晰的自然语言 + 必要的上下文
  - 可以先让 AI 生成 TODO list 或者实现方案，和它讨论几次确认最终方案之后再实际做代码工作
4. 审查生成代码：理解它做了什么，是否正确
5. 测试验证：跑通测试，手动验证边界情况
6. 迭代修正：不对就给反馈，让 AI 修改
```

核心原则：**你是驾驶员，AI 是副驾驶。**

## 一些 prompt 对比

- **清晰描述需求和约束**
  - 应当："用 React + TypeScript 写一个待办事项组件，支持添加和删除"
  - 不应当："帮我写个前端"
- **理解 AI 生成的每一块代码**
  - 应当："这段代码在做什么？为什么用这个 API？"
  - 不应当在疑惑的时候什么都不说
- **拆分复杂任务**
  - 应当：先"帮我生成一个聊天框，样式可以参考这张图片"，然后"帮我给输入框绑定一个处理函数，可以点击确认发消息"
  - 不应当："帮我写个聊天界面，可以收发消息，消息可以回复和撤回"
- **给模糊的 Prompt**
  - 应当："这个列表渲染有 1s 延迟，请用 useMemo 优化重渲染"
  - 不应当："帮我优化一下速度"

## 一些习惯

- **善用 AI 做重复性工作**
  - 样板代码、类型定义、单元测试、文档注释
- **保持版本管理习惯**
  - 每次 AI 生成的改动，review 后再 commit
- **不要盲目接受不理解的代码**
  - AI 会自信地生成看似正确但有 bug 的代码
- **一定审查安全关键逻辑**
  - 认证、权限、数据校验——这些必须你自己把关
- **不要跳过测试**
  - AI 生成的代码同样需要测试，甚至更需要
- **不要放弃基本功**
  - 不理解组件、状态、异步这些概念，你就无法有效审查 AI 的输出
  - Vibe coding 降低了写代码的门槛，但**提高了对判断力的要求**

## 对本课程的启示

| 以前 | 现在 |
|------|------|
| 语法要背 | 语法让 AI 查 |
| API 要记 | API 让 AI 写 |
| 概念可以模糊 | **概念必须你掌握** |
| 架构跟着教程走 | **架构由你决定** |

你需要真正理解的是：
- 组件是什么？Props 和 State 的区别？
- 为什么需要异步？事件循环是怎么回事？
- 副作用是什么？useEffect 解决什么问题？

这些概念决定了你能否**提出正确的问题**，以及能否**判断 AI 的回答是否靠谱**。
